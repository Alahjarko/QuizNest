# QuizNest 项目开发规则与产品愿景 (Workspace Rules)

本规则规定了 QuizNest 的定位、产品主线、界面美学设计与开发纪律，后续所有迭代和修改必须严格遵循本指南。

---

## 1. 产品定位与核心闭环

* **定位**：本地运行的 AI 助学软件，目标用户主要是理科大学生、考研学生以及需要长期复习复杂知识体系的人。
* **核心价值**：把用户自己的笔记、课件、PDF、PPT 转化成可练习、可复盘、可追问的学习系统。
* **核心业务闭环**：
  $$\text{笔记/课件} \rightarrow \text{AI 整理} \rightarrow \text{生成题组} \rightarrow \text{用户答题} \rightarrow \text{AI 判题与讲解} \rightarrow \text{错题本} \rightarrow \text{解惑追问} \rightarrow \text{薄弱点复习} \rightarrow \text{再练习}$$
* **基本原则**：控制功能堆砌，以今日学习的下一步清晰动作（继续练什么、复习错题、复习章节、追问解惑）为引导。

---

## 2. 技术栈与开发主线

* **技术栈**：Tauri v2 + 原生 JavaScript (无 UI 框架，ESM) + 原生 CSS + IndexedDB。
* **主线任务**：继续维护 Tauri 桌面版本。**严禁重构、更改或补齐曾经废弃的 native SwiftUI 探索版本**。
* **打包策略**：macOS Apple Silicon 采用本地打包路线（`npm run build:dmg`），不走 GitHub 云端打包，产物命名格式为 `QuizNest_0.1.8_aarch64.dmg`。

---

## 3. 产品气质与审美方向

* **产品气质**：**Academic Problem Solver + Personal Study Desk**。追求学术、克制、安静、可靠、求解感。禁止将其改造成花哨的 AI SaaS、培训网课风或游戏化学习软件。
* **UI 与配色**：
  * 左侧应用级折叠侧边栏，折叠切换采用**纯网格平滑过渡**（B 方案），严禁使用易导致残影与闪烁的 View Transitions API 快照动画。
  * 浅色、低饱和度、纸面感材质。
  * **低饱和绿色**（如 `--accent: #2d6b52`）作为主强调色。
  * 卡片采用轻边框、轻阴影、小圆角设计。
  * 首页头图应设计为“个人学习封面”，避免过重的宣传 Banner。
  * 动画克制、短促，禁止花哨的渐变、弹跳和玻璃拟态。
  * **必须尊重并适配** `prefers-reduced-motion`。

---

## 4. 交互动效时间规范 (CSS / JS)

动效必须严格服务于状态的变化，时间建议遵循以下规范：
* **Hover / Press 状态反馈**：$100\text{ms} - 140\text{ms}$
* **卡片悬浮 (Hover Floating)**：$120\text{ms} - 180\text{ms}$
* **页面内容切入 (Page Transition)**：$140\text{ms} - 180\text{ms}$
* **Toast / 保存成功反馈 (如“已保存 ✓”)**：$150\text{ms} - 220\text{ms}$
* **侧边栏折叠**：保持现有的纯网格过渡逻辑，不做额外改动。

---

## 5. 核心页面设计规范

### 1. 首页 (今日学习桌面)
* 像“今日学习驾驶舱”，告诉用户下一步该做什么（继续练习、复习错题、生成题组、展示连续学习热力图与复习建议），绝非营销页面或纯堆砌数据的统计板。

### 2. 题组库 (问题集档案 - Problem Sets)
* 卡片突出题组名、所属笔记、状态（未开始/进行中/已完成/建议复习）、正确率和主行动按钮。
* 每张卡片**仅保留一个最关键主操作**，重命名、删除等次要功能全部收纳进更多菜单（`OverflowMenu`）。

### 3. 错题本 (错题诊断手册)
* 默认仅以摘要展示错因、所属章节、知识点和掌握状态。完整题目、答案与 AI 判题解析默认折叠，按需点击展开，防止用户信息过载。

### 4. 解惑 (带上下文的解题工作台)
* 坚守“上下文优先”，清晰标明当前提问所绑定的笔记、题组、题目和错题。优化 Context Chips、右侧面板和快捷追问（如“类似题”、“简单解释”、“检查理解”）。

### 5. 统计页 (学习洞察报告)
* 降低 Token 消耗等技术指标权重，突出 7 天学习热力图、错题高发章节、低正确率题型以及“下一步复习建议”。

---

## 6. 开发纪律与提交流程

1. 严禁无关联的大规模重构。
2. 严禁破坏 IndexedDB 数据表结构，保证向后兼容性。
3. 严格禁止硬编码虚假统计数据。
4. 任何提交流程前必须在本地依次运行验证：
   * `npm run sync-dist`（同步资源）
   * JavaScript 语法检查 (`node --check`)
   * `git diff --check`（无冲突和空白行残留）
5. **Git 提交纪律**：任何 commit、push 操作必须在用户明确指示下进行，严禁擅自提交。
6. **验收标准**：每次完成新的修改任务后，必须使用 `agent-browser` 技能亲自启动开发环境并打开软件，对自己刚刚做好的功能进行调试，确认视觉与功能完全没有问题后才算验收成功。
