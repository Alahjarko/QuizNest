# ⚠️ Before You Write the Feature Log

在向 `FEATURE_LOG.md` 写入新的工作记录之前，请所有 AI 助手严格遵守以下规范，以保持协作日志的一致性和可读性：

## 1. 绝对的时间倒序（Descending Order）
- **永远将最新的功能记录写在最前面**。
- 将你的更新插入在 `## 完整更新记录` 标题下的**最顶端**。
- 请不要像过去那样把记录追加到文件末尾。

## 2. 统一的时间线（Unified Timeline）
- 不要再为了不同的 AI（如 Codex、GLM 等）创建单独的 Header 区块（例如 `## GLM 已完成的功能`）。
- 所有的更新都必须放置在同一个大区块 `## 完整更新记录` 下。
- 你可以直接在你的单条记录标题中标明你的身份。
- **标题格式参考**：`### YYYY-MM-DD / [你的名字] / [使用的模型]`
  - 例如：`### 2026-06-22 / Antigravity CLI / Gemini 3.1 Pro`

## 3. 干扰信息沉底（Keep the Timeline Clean）
- 如果有“后续建议”、“给下一个 AI 的备忘录（Notes/Memos）”等非功能迭代性质的内容，**严禁夹杂在时间线的记录中**。
- 请将这些内容统一放置到文件的最底部（如 `## 后续建议` 或 `---` 之后）。

## 4. 必填字段
在你的记录正文中，必须清晰列出：
- **完成内容**：你做了哪些代码改动、解决了什么问题。
- **涉及文件**：列出被修改或新增的核心文件路径。
- **验证方式**：说明你是如何确保代码不崩溃的（比如通过 `npm run build`、`node --check` 或是自动化打包测试等）。

---
*遵守以上准则，保持时间线的整洁，让每一位接手的智能体都能以最高效的方式回顾项目进展。*
